Short answer: There is no single “quick” automatic technique that takes an Ada grammar and directly produces a DIANA‑style structure. DIANA is not generated from the grammar alone; it is defined as an IDL‑specified abstract data type that must be manually mapped to the grammar’s constructs. Wikipedia. DIANA (intermediate language) - Wikipedia
Why there is no direct grammar → DIANA pipeline

DIANA (Descriptive Intermediate Attributed Notation for Ada) was designed as a semantic intermediate representation, not as a syntax‑driven AST generator. It is defined using IDL, which describes the data structures and attributes that represent Ada semantics. DIANA’s purpose is to sit between the front end (parsing + semantic analysis) and the back end. Wikipedia. DIANA (intermediate language) - Wikipedia

Because of this design:

    DIANA nodes correspond to semantic concepts, not grammar productions.

    The Ada grammar alone does not contain enough information to construct DIANA structures.

    The mapping must be written manually in the compiler front end: the parser builds a syntax tree, and semantic actions populate DIANA nodes.

This is why DIANA is described as an attributed abstract syntax tree enriched with semantic information. Springer. The use of Diana in compilers, language transformers ... - Springer
What metacompilers can do

Metacompilers (e.g., META‑II, attribute‑grammar systems, operational‑semantics‑based systems) can generate:

    parsers from grammars,

    AST constructors,

    attribute evaluators,

    semantic analyzers.

But DIANA is not an attribute grammar; it is a fixed, hand‑designed IR. Metacompilers can help build a compiler that outputs DIANA, but they cannot infer DIANA from the grammar. This is consistent with research showing that metacompilers rely on explicit semantic specifications (attribute grammars, operational semantics, etc.) rather than deriving semantics from syntax alone. ResearchGate. A comparison of two metacompilation approaches to implementing a ...
What is the quickest practical workflow?

Although no automatic grammar→DIANA generator exists, the fastest practical approach is:

    Start with the Ada grammar (BNF or equivalent).

    Use a metacompiler or parser generator to produce:

        a parser,

        an initial AST.

    Manually define the mapping from AST nodes to DIANA nodes:

        For each grammar production, specify semantic actions that construct the appropriate DIANA IDL‑defined structures.

    Use the DIANA IDL specification to generate the DIANA data types in your implementation language. Wikipedia. DIANA (intermediate language) - Wikipedia

    Implement attribute propagation (DIANA’s “attribution” model) to fill in semantic information. Springer. The use of Diana in compilers, language transformers ... - Springer

This is the same approach used historically in Ada compilers: the grammar drives parsing, but DIANA is built by explicit semantic routines.
Why this cannot be automated from grammar alone

A grammar describes syntax. DIANA describes semantics. The gap between them is large:

    DIANA nodes encode type information, overload resolution, visibility, symbol tables, etc.

    None of this is present in the grammar.

    Therefore, no tool can infer DIANA structures from grammar without additional semantic specifications.

This is why DIANA is defined in IDL rather than being generated from the Ada grammar.
