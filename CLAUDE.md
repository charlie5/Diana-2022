# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this project is

The goal is to design an updated, **backwards-compatible DIANA intermediate representation for Ada 2022** (`DIANA_2022`), plus an **interpretive test harness** that can execute a given DIANA tree. DIANA is a *semantic* IR (an attributed abstract syntax tree with cross-links), not a parse tree ŌĆö nodes represent meaningful Ada constructs, not grammar productions.

## Current state (read before assuming structure)

The **spec is substantially complete** and the **interpretive harness has been started** (a working first increment). Key locations:

- `spec/DIANA_2022.idl` ŌĆö the canonical IR spec (single-parent partitioned, referentially complete; verified by the tools below).
- `spec/DIANA_2022_Compatibility.idl` ŌĆö the Rev 3/4 `Renames` alias layer.

## Building & running the harness

The harness is an Ada 2022 / GNAT project under `harness/`. The root `Makefile`
wires the whole pipeline; **`make check`** is the one-command verification (spec
invariants ŌåÆ generated files in sync with the spec ŌåÆ `gprbuild` ŌåÆ run both
demos), and is the thing to run after any change:

```sh
make check    # spec checkers + generated-file sync + build + run both demos
make build    # just gprbuild (both demos)
make run      # run harness/diana_harness and harness/interp_demo
make gen      # re-generate diana-nodes/builders/accessors after a spec change
```

Or directly:

```sh
gprbuild -P harness/diana_harness.gpr   # build both demos (object dir harness/obj)
harness/diana_harness                   # the separate-compilation + builder/accessor demo
harness/interp_demo                     # build a small program and execute it (the interpreter)
```

`make check`'s `verify-gen` re-runs the generators and diffs against the
committed `diana-nodes.ads` / `diana-builders.ads` / `diana-accessors.ads`, so a
spec edit without `make gen` (or a hand-edit of a generated file) fails the
check.

Architecture (decided with Shark8 ŌĆö see the header of `harness/src/diana.ads`):
- The whole program library is **one** `Ada.Containers.Indefinite_Multiway_Trees` of `Diana.Node'Class`. Parent/child edges are the *structural* attributes; the tree root's children are per-compilation subtrees.
- IDL classes become abstract intermediate tagged types; concrete node kinds are leaves (single inheritance == the single-parent partition). "Is X an Expression?" is just `X in Expression'Class`.
- Semantic cross-links are tree **`Cursor`s** (the in-memory form of a symbolic external label).
- A `with`'ed-but-unloaded unit is a `Pending_Unit` **stub** child of the root that records its inbound `Referrer`s. `Diana.Library.Merge` replaces the stub with the real compilation and **fixes up those referrers in place**; `Require_All_Resolved` raises `Missing_Compilation` if any stub remains (the "error out on missing separate compilation" requirement). `Merge`'s `Kind` selects what the merged unit declares ŌĆö a plain object (`Object_Unit`), a package (`Package_Unit`, a `Package_Name`), or a generic package template (`Generic_Package_Unit`, a `Generic_Name` with a `Package_Specification`), so an **instantiation of a separately-compiled generic package** resolves to the generic when its compilation arrives. `Merge`'s `Parent` (non-empty) makes the merged unit a **child unit**, loaded beneath its parent's compilation in the forest and requiring the parent be compiled first (else `Missing_Compilation`) ŌĆö `Find_Unit` searches the whole forest so a nested child is still found. A **child-generic instantiation** names the generic by a compound name (a `Selected_Component`), so it carries two referrers ŌĆö the prefix (the parent) and the selector (the child) ŌĆö each fixed up by merging the respective unit. The parent of a child may itself be a separately-compiled generic package (a **child of a generic** is implicitly generic); it is then used through a parent *instance* (`package I is new Parent; package C is new I.Child;`), so the child instantiation's prefix resolves locally to the instance and only its selector is the cross-unit reference ŌĆö the existing `Parent`/`Generic_Package_Unit` machinery composes for this with no special case. The hierarchy nests to arbitrary depth: a **grandchild** `A.B.C` is merged with `Parent => "A.B"` (the recursive `Find_Unit` locates a child nested any number of levels deep), named by a doubly-nested `Selected_Component` `(A.B).C` carrying one referrer per ancestor level, with the whole chain `A ŌåÉ A.B ŌåÉ A.B.C` enforced top-down. `Merge`'s `Nested` (non-empty) instead declares a generic package **nested inside** the merged unit's specification ŌĆö *not* a separate compilation ŌĆö so a **nested-generic instantiation** (`Outer.Inner`) records both its prefix and its selector against the *same* unit (`Outer`) and a single merge resolves both (the re-target loop maps each referrer to the unit's entity or the nested generic by its wanted name). `Merge_Subunit` handles **subunits** (`is separate` bodies): rather than resolving a *name* (`Used_Name.Definition`), it completes a *body stub* ŌĆö re-targeting each recorded referrer's `Subprogram_Body.Completion` from its `Stub` to the subunit's proper body, loaded beneath the **enclosing compilation** (which carries the stub and so must already be compiled). `Merge_Subunit`'s `Enclosing` parameter (default = `Parent`) distinguishes the program unit the subunit *completes* from the loaded compilation that *carries* the stub: for a subunit of a **nested generic** (`Sessions.Pool.Evict`) the parent unit (`Sessions.Pool`) is declared inside another compilation and is not a unit of its own, so `Enclosing` names that compilation (`Sessions`).
- The full node set (`Diana.Nodes`) is **generated from `spec/DIANA_2022.idl`** by `tools/gen_nodes.pl` (436 tagged types), with a builder/accessor API (`Diana.Builders`, `Diana.Accessors`) generated by `tools/gen_api.pl`. Build trees bottom-up with `Diana.Builders.<Node> (...)` + `Add_Child`; read them with `Diana.Accessors.As_<T> (C).Attribute`. The merge/stub bookkeeping types that are *not* part of the IR (`Loaded_Unit`, `Pending_Unit`, `Referrer`) live in the hand-written `Diana.Loading`.

## The IDL dialect ŌĆö this is a frequent mistake

The IR is specified in the **Wulf/Lamb/Nestor IDL** (an abstract-data-type / specification language), **not CORBA IDL**. Do not write OO `interface { field; }` syntax. The notation below is taken from the Rev 3 manual ┬¦1.4 (`references/DIANA_Rev3_ADA128232.pdf`, doc pp. 21ŌĆō30):

- Reserved words are capitalized. The whole IR is wrapped in one structure: `Structure DIANA_2022 Root Compilation_Unit Is  -- rules  End`.
- **Class** (`::=`): `EXP ::= leaf | tree ;` ŌĆö the RHS is *only* an alternation of class/node names (no concatenation of items). Class names defined via `::=` must be acyclic (form a DAG). Extra `::=` rules for the same class add alternatives. Class names never appear in actual trees; only node names do.
- **Node** (`=>`): `tree => op: OPERATOR, left: EXP, right: EXP ;` ŌĆö attributes are `name: TYPE`, comma-separated, **order-independent**. Extra `=>` rules for the same node *add* attributes (they are not alternatives). An attribute-less node `void => ;` acts like an enumeration literal.
- **Attribute types:** the basic types `Boolean`, `Integer`, `Rational`, `String`, `Seq Of T`; a `<name>` reference to another node or class; or a **private type** declared `Type Source_Position;` (used for implementation-specific data like source positions).
- **Sequence idiom:** a name ending in `_S` (class) / `_s` (node) is always a sequence of what precedes the `_` ŌĆö e.g. `FOO_S => foo_s ;` then `foo_s => as_list: Seq Of FOO ;`.
- **Optional idiom:** a class ending in `_VOID` is always `FOO_VOID => FOO | void ;` with `void => ;`.
- **Refinement / renaming:** IDL has native `Structure X Refines Y Is ŌĆ” End` and `Structure X Renames Y Is ŌĆ” End` forms, plus representation clauses `For MyType Use MyPackage;` (internal) and `For MyType Use External ExternType;` (external).
- Discriminated Ada records map to a class: the discriminant is an attribute on the parent class, each variant is a node type, a `null` variant is an attribute-less node type.

### The four attribute prefixes (DIANA's attribute classification)

Every DIANA attribute is prefixed to mark which of four kinds it is (Rev 3 ┬¦1.4.4 / ┬¦1.3):

- `as_` ŌĆö **structural** (defines the abstract syntax tree; names mirror the Ada Formal Definition, prefixed with `as_`).
- `lx_` ŌĆö **lexical** (source form: identifier spelling, source position ŌĆö e.g. `lx_srcpos`).
- `sm_` ŌĆö **semantic** (results of semantic analysis: type, overload resolution ŌĆö e.g. `sm_type_spec`).
- `cd_` ŌĆö **code** (there is essentially one: representation info the back end must observe).

Convention is to list a node's attributes grouped structural ŌåÆ lexical ŌåÆ semantic ŌåÆ code, in that order.

## Style rules (from `Initial_instructions.md`)

- **Identifiers:** `Camel_Case` with underscores ŌĆö initial capitals, underscore-separated, **full words, no abbreviations** (`Object_Declaration`, not `object_decl`; `Subprogram_Body`, not `subp_body`). All-capital acronyms stay capitalized.
- **Provenance comments:** annotate constructs with where they come from ŌĆö DIANA RM vs the Ada RM. Rev 3 already did this: each rule group opens with a comment citing the corresponding Ada LRM section. Mirror that convention (citing the Ada 2022 RM), and add a DIANA-RM citation where a node descends from the historical design.
- Expose the shortened DIANA Rev 3/4 node names via **`RENAMES`** ŌĆö use IDL's native `Structure ŌĆ” Renames ŌĆ” Is ŌĆ” End` form for this; the full-word `Camel_Case` names are canonical.

Note the **naming-style shift from original DIANA**: Rev 3/4 use ALL-UPPER-CASE class names (`DECL`, `EXP`, `STM`, `TYPE_SPEC`, `DEF`), all-lower-case node names (`tree`, `constant`, `var`), and prefixed lower-case attributes (`as_exp`, `sm_type_spec`, `lx_srcpos`). DIANA_2022's canonical names are `Camel_Case` full words instead; the historical short/lower-case names live behind the `Renames` compatibility layer.

**Attribute naming (decided):** DIANA_2022 uses **no attribute prefixes** ŌĆö attributes are named purely by meaning (`Object_Type`, `Source_Position`, `Definition`), not `as_`/`lx_`/`sm_`/`cd_`. The four-way structural/lexical/semantic/code *classification* is still preserved, but shown by grouping and comments within each node, not encoded in the identifier.

**Structure (decided):** follow Rev 4's approach ŌĆö a single-parent class hierarchy (each node/class in at most one class) with shared attributes **hoisted** to the highest applicable class. Universal lexical attributes (`Source_Position`, `Comments`) live on the root node class rather than being repeated per node.

**`.idl` layout (decided):** one attribute/alternative per line with aligned colons; the terminating `;` attaches to the **last element** (never on its own line); trailing kind/provenance comments aligned within each node. The spec lives in `spec/DIANA_2022.idl`.

## Core design decisions (non-obvious; keep consistent)

These are deliberate modeling choices that span the whole IR ŌĆö preserve them:

- **Backwards compatibility** with the original DIANA (Rev 3/4) is a hard requirement.
- **Collapse semantically-equivalent Ada forms into one node.** Example: `return 0;`, `return Value : Integer := 0;`, and the full extended return are all one return construct.
- **Aspects and attribute-definition clauses unify** into a single `Semantic_Property` node (`with Size => 8` and `for T'Size use 8;` are the same semantic property). The node always carries a `Source_Position` and an `Origin` note (`From_Aspect | From_Attribute_Clause`) recording the syntactic source. Attach semantic properties to the declarations/types/subprograms/objects that can legally carry them.
- **The generic formal system mirrors the type system** ŌĆö formal scalar/composite/access/interface/private types parallel their non-formal counterparts so actuals line up structurally with formals.
- **Name binding is the symbol table.** Each declared entity has one *defining occurrence* (`DEF_ID` node) holding all its semantic attributes; *used occurrences* point back via `sm_defn` (`sm_operator` for built-in ops). Multiple declaration points (incomplete/private types, deferred constants, spec+body) are linked by `sm_first`. The interpretive harness's name resolution and its merging of separately-compiled units both hinge on this `sm_defn`/`sm_first` model. Calls and overloadable operators share one `function_call` node (with `lx_prefix` recording infix vs. prefix).

## Test harness requirements

The interpretive harness must:
- Execute a given DIANA tree, and **error out if execution cannot be completed** ŌĆö demonstrated two ways: a missing separate compilation (`Require_All_Resolved` / `Missing_Compilation`) and an unexecutable tree (`Diana.Interpreter` / `Interpretation_Error`, e.g. an unbound variable) or a failed runtime contract (`Assertion_Error`, e.g. a `pragma Assert` or a `Pre`/`Post`).
- Provide a way to **merge in separately-compiled DIANA** ŌĆö demonstrated by `Diana.Library.Merge`.

`Diana.Interpreter` is a first execution slice: it walks a tree through the accessors and evaluates it over the `Static_Value` model ŌĆö integer / real / boolean / string literals, **access values** (`new T'(x)` allocates onto a heap, `.all` dereferences, `null`, `X.all := v`, access equality/aliasing), **array / record aggregates** with composite value semantics (`(1, 2, 3)` / `(X => 1, Y => 2)`, indexing `A (I)`, selection `R.F`, component assignment, deep copy on `B := A`, and the `'First`/`'Last`/`'Length` array attributes), variable references, built-in operator calls (integer & real arithmetic with promotion, comparison over numbers/strings, string `&` concatenation, boolean ops), assignment, if-statements, while-loops, **`for` loops** ŌĆö range-based (`for I in L .. H`) and container (`for E of array`, or over a record's components in component-name order), both with `reverse` and the Ada 2022 `when` iterator filter, **case statements** (value / `1 | 2` / `3 .. 4` range / `others` choices; Generic instantiation is executed by `Invoke`: a `Subprogram_Name` whose `Completion` is a `Unit_Instantiation` is treated as a call into its template ŌĆö `Invoke` resolves the `Generic_Name`, runs that template's body with the generic formal objects bound (in the call scope) to the instantiation's actuals, alongside the call's own parameters.

A **generic package** is a `Generic_Name` whose `Specification` is a `Package_Specification` (rather than a `Generic_Subprogram_Header`). `Elaborate` instantiates one when it meets a `Package_Declaration` whose `Completion` is a `Unit_Instantiation`: `Instantiate_Package` creates a fresh **instance scope** (parented to the declaring scope, so a nested instance sees enclosing names), binds the generic formal objects to the actuals, elaborates the template's visible declarations into that scope, and binds the instance name to a `Package_Value` (a new `Static_Value` kind that is a runtime-only handle holding the instance scope's index). `Pkg.Member` resolves by looking the selector up in that scope; `Pkg.Sub (...)` calls the member subprogram with the instance scope as its static link (passed to `Invoke` as `Home`), so a member closes over the package's own generic formals and declarations.

A **child generic package** is instantiated against a parent *instance* ŌĆö its `Generic_Instantiation.Generic_Unit` is a `Selected_Component` (`Parent_Instance.Child`) rather than a plain name. `Instantiate_Package` then nests the child's instance scope in the **parent instance's** scope (its `Home_Scope`) instead of the lexical declaring scope, so the child's declarations and member bodies see the parent's members by simple name (and the static-link walk from a child member reaches the parent's subprograms), exactly as a real child unit sees its parent's visible part.
