# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this project is

The goal is to design an updated, **backwards-compatible DIANA intermediate representation for Ada 2022** (`DIANA_2022`), plus an **interpretive test harness** that can execute a given DIANA tree. DIANA is a *semantic* IR (an attributed abstract syntax tree with cross-links), not a parse tree — nodes represent meaningful Ada constructs, not grammar productions.

## Building & running the harness

The harness is an Ada 2022 / GNAT project under `harness/`. The root `Makefile`
wires the whole pipeline; **`make check`** is the one-command verification (spec
invariants → generated files in sync with the spec → `gprbuild` → run both
demos), and is the thing to run after any change:

```sh
make check    # spec checkers + generated-file sync + build + run both demos
make build    # just gprbuild (both demos)
make run      # run harness/diana_harness and harness/interp_demo
make gen      # re-generate diana-nodes/builders/accessors after a spec change
```

Or directly:

```sh
gprbuild -P harness/diana_harness.gpr   # build both demos (object dir harness/obj)
harness/diana_harness                   # the separate-compilation + builder/accessor demo
harness/interp_demo                     # build a small program and execute it (the interpreter)
```

`make check`'s `verify-gen` re-runs the generators and diffs against the
committed `diana-nodes.ads` / `diana-builders.ads` / `diana-accessors.ads`, so a
spec edit without `make gen` (or a hand-edit of a generated file) fails the
check.

Architecture (decided with Shark8 — see the header of `harness/src/diana.ads`):
- The whole program library is **one** `Ada.Containers.Indefinite_Multiway_Trees` of `Diana.Node'Class`. Parent/child edges are the *structural* attributes; the tree root's children are per-compilation subtrees.
- IDL classes become abstract intermediate tagged types; concrete node kinds are leaves (single inheritance == the single-parent partition). "Is X an Expression?" is just `X in Expression'Class`.
- Semantic cross-links are tree **`Cursor`s** (the in-memory form of a symbolic external label).
- A `with`'ed-but-unloaded unit is a `Pending_Unit` **stub** child of the root that records its inbound `Referrer`s. `Diana.Library.Merge` replaces the stub with the real compilation and **fixes up those referrers in place**; `Require_All_Resolved` raises `Missing_Compilation` if any stub remains (the "error out on missing separate compilation" requirement). `Merge`'s `Kind` selects what the merged unit declares — a plain object (`Object_Unit`), a package (`Package_Unit`, a `Package_Name`), or a generic package template (`Generic_Package_Unit`, a `Generic_Name` with a `Package_Specification`), so an **instantiation of a separately-compiled generic package** resolves to the generic when its compilation arrives. `Merge`'s `Parent` (non-empty) makes the merged unit a **child unit**, loaded beneath its parent's compilation in the forest and requiring the parent be compiled first (else `Missing_Compilation`) — `Find_Unit` searches the whole forest so a nested child is still found. A **child-generic instantiation** names the generic by a compound name (a `Selected_Component`), so it carries two referrers — the prefix (the parent) and the selector (the child) — each fixed up by merging the respective unit. The parent of a child may itself be a separately-compiled generic package (a **child of a generic** is implicitly generic); it is then used through a parent *instance* (`package I is new Parent; package C is new I.Child;`), so the child instantiation's prefix resolves locally to the instance and only its selector is the cross-unit reference — the existing `Parent`/`Generic_Package_Unit` machinery composes for this with no special case. The hierarchy nests to arbitrary depth: a **grandchild** `A.B.C` is merged with `Parent => "A.B"` (the recursive `Find_Unit` locates a child nested any number of levels deep), named by a doubly-nested `Selected_Component` `(A.B).C` carrying one referrer per ancestor level, with the whole chain `A ← A.B ← A.B.C` enforced top-down. `Merge`'s `Nested` (non-empty) instead declares a generic package **nested inside** the merged unit's specification — *not* a separate compilation — so a **nested-generic instantiation** (`Outer.Inner`) records both its prefix and its selector against the *same* unit (`Outer`) and a single merge resolves both (the re-target loop maps each referrer to the unit's entity or the nested generic by its wanted name). `Merge_Subunit` handles **subunits** (`is separate` bodies): rather than resolving a *name* (`Used_Name.Definition`), it completes a *body stub* — re-targeting each recorded referrer's `Subprogram_Body.Completion` from its `Stub` to the subunit's proper body, loaded beneath the **enclosing compilation** (which carries the stub and so must already be compiled). `Merge_Subunit`'s `Enclosing` parameter (default = `Parent`) distinguishes the program unit the subunit *completes* from the loaded compilation that *carries* the stub: for a subunit of a **nested generic** (`Sessions.Pool.Evict`) the parent unit (`Sessions.Pool`) is declared inside another compilation and is not a unit of its own, so `Enclosing` names that compilation (`Sessions`).
- The full node set (`Diana.Nodes`) is **generated from `spec/DIANA_2022.idl`** by `tools/gen_nodes.pl` (436 tagged types), with a builder/accessor API (`Diana.Builders`, `Diana.Accessors`) generated by `tools/gen_api.pl`. Build trees bottom-up with `Diana.Builders.<Node> (...)` + `Add_Child`; read them with `Diana.Accessors.As_<T> (C).Attribute`. The merge/stub bookkeeping types that are *not* part of the IR (`Loaded_Unit`, `Pending_Unit`, `Referrer`) live in the hand-written `Diana.Loading`.

## The IDL dialect — this is a frequent mistake

The IR is specified in the **Wulf/Lamb/Nestor IDL** (an abstract-data-type / specification language), **not CORBA IDL**. Do not write OO `interface { field; }` syntax. The notation below is taken from the Rev 3 manual §1.4 (`references/DIANA_Rev3_ADA128232.pdf`, doc pp. 21–30):

- Reserved words are capitalized. The whole IR is wrapped in one structure: `Structure DIANA_2022 Root Compilation_Unit Is  -- rules  End`.
- **Class** (`::=`): `EXP ::= leaf | tree ;` — the RHS is *only* an alternation of class/node names (no concatenation of items). Class names defined via `::=` must be acyclic (form a DAG). Extra `::=` rules for the same class add alternatives. Class names never appear in actual trees; only node names do.
- **Node** (`=>`): `tree => op: OPERATOR, left: EXP, right: EXP ;` — attributes are `name: TYPE`, comma-separated, **order-independent**. Extra `=>` rules for the same node *add* attributes (they are not alternatives). An attribute-less node `void => ;` acts like an enumeration literal.
- **Attribute types:** the basic types `Boolean`, `Integer`, `Rational`, `String`, `Seq Of T`; a `<name>` reference to another node or class; or a **private type** declared `Type Source_Position;` (used for implementation-specific data like source positions).
- **Sequence idiom:** a name ending in `_S` (class) / `_s` (node) is always a sequence of what precedes the `_` — e.g. `FOO_S => foo_s ;` then `foo_s => as_list: Seq Of FOO ;`.
- **Optional idiom:** a class ending in `_VOID` is always `FOO_VOID => FOO | void ;` with `void => ;`.
- **Refinement / renaming:** IDL has native `Structure X Refines Y Is … End` and `Structure X Renames Y Is … End` forms, plus representation clauses `For MyType Use MyPackage;` (internal) and `For MyType Use External ExternType;` (external).
- Discriminated Ada records map to a class: the discriminant is an attribute on the parent class, each variant is a node type, a `null` variant is an attribute-less node type.

### The four attribute prefixes (DIANA's attribute classification)

Every DIANA attribute is prefixed to mark which of four kinds it is (Rev 3 §1.4.4 / §1.3):

- `as_` — **structural** (defines the abstract syntax tree; names mirror the Ada Formal Definition, prefixed with `as_`).
- `lx_` — **lexical** (source form: identifier spelling, source position — e.g. `lx_srcpos`).
- `sm_` — **semantic** (results of semantic analysis: type, overload resolution — e.g. `sm_type_spec`).
- `cd_` — **code** (there is essentially one: representation info the back end must observe).

Convention is to list a node's attributes grouped structural → lexical → semantic → code, in that order.

## Style rules (from `Initial_instructions.md`)

- **Identifiers:** `Camel_Case` with underscores — initial capitals, underscore-separated, **full words, no abbreviations** (`Object_Declaration`, not `object_decl`; `Subprogram_Body`, not `subp_body`). All-capital acronyms stay capitalized.
- **Provenance comments:** annotate constructs with where they come from — DIANA RM vs the Ada RM. Rev 3 already did this: each rule group opens with a comment citing the corresponding Ada LRM section. Mirror that convention (citing the Ada 2022 RM), and add a DIANA-RM citation where a node descends from the historical design.
- Expose the shortened DIANA Rev 3/4 node names via **`RENAMES`** — use IDL's native `Structure … Renames … Is … End` form for this; the full-word `Camel_Case` names are canonical.

Note the **naming-style shift from original DIANA**: Rev 3/4 use ALL-UPPER-CASE class names (`DECL`, `EXP`, `STM`, `TYPE_SPEC`, `DEF`), all-lower-case node names (`tree`, `constant`, `var`), and prefixed lower-case attributes (`as_exp`, `sm_type_spec`, `lx_srcpos`). DIANA_2022's canonical names are `Camel_Case` full words instead; the historical short/lower-case names live behind the `Renames` compatibility layer.

**Attribute naming (decided):** DIANA_2022 uses **no attribute prefixes** — attributes are named purely by meaning (`Object_Type`, `Source_Position`, `Definition`), not `as_`/`lx_`/`sm_`/`cd_`. The four-way structural/lexical/semantic/code *classification* is still preserved, but shown by grouping and comments within each node, not encoded in the identifier.

**Structure (decided):** follow Rev 4's approach — a single-parent class hierarchy (each node/class in at most one class) with shared attributes **hoisted** to the highest applicable class. Universal lexical attributes (`Source_Position`, `Comments`) live on the root node class rather than being repeated per node.

**`.idl` layout (decided):** one attribute/alternative per line with aligned colons; the terminating `;` attaches to the **last element** (never on its own line); trailing kind/provenance comments aligned within each node. The spec lives in `spec/DIANA_2022.idl`.

## Core design decisions (non-obvious; keep consistent)

These are deliberate modeling choices that span the whole IR — preserve them:

- **Backwards compatibility** with the original DIANA (Rev 3/4) is a hard requirement.
- **Collapse semantically-equivalent Ada forms into one node.** Example: `return 0;`, `return Value : Integer := 0;`, and the full extended return are all one return construct.
- **Aspects and attribute-definition clauses unify** into a single `Semantic_Property` node (`with Size => 8` and `for T'Size use 8;` are the same semantic property). The node always carries a `Source_Position` and an `Origin` note (`From_Aspect | From_Attribute_Clause`) recording the syntactic source. Attach semantic properties to the declarations/types/subprograms/objects that can legally carry them.
- **The generic formal system mirrors the type system** — formal scalar/composite/access/interface/private types parallel their non-formal counterparts so actuals line up structurally with formals.
- **Name binding is the symbol table.** Each declared entity has one *defining occurrence* (`DEF_ID` node) holding all its semantic attributes; *used occurrences* point back via `sm_defn` (`sm_operator` for built-in ops). Multiple declaration points (incomplete/private types, deferred constants, spec+body) are linked by `sm_first`. The interpretive harness's name resolution and its merging of separately-compiled units both hinge on this `sm_defn`/`sm_first` model. Calls and overloadable operators share one `function_call` node (with `lx_prefix` recording infix vs. prefix).

Generic instantiation is executed by `Invoke`: a `Subprogram_Name` whose `Completion` is a `Unit_Instantiation` is treated as a call into its template — `Invoke` resolves the `Generic_Name`, runs that template's body with the generic formals bound (in the call scope) to the instantiation's actuals, alongside the call's own parameters. A formal **object** binds to the actual's value; a formal **subprogram** binds (in the scope's `Formal_Subs` map, keyed by the formal's name → the actual subprogram's defining occurrence) so that a call to the formal inside the template dispatches to the actual (the function-call path checks `Lookup_Formal_Sub` before the ordinary subprogram path); a formal **type** is erased — it binds nothing and only consumes its positional actual slot, since `Static_Value` is dynamically typed, so one template body runs over whatever value kinds the actuals supply (this covers a formal **interface** type — its operations ride in as formal subprograms, the "program to an interface" pattern — a formal **derived** type, whose parent's inherited operations the body simply uses directly on the values; a formal **scalar** type (`range <>` etc.), whose ordering/arithmetic are the interpreter's built-in operations on the values; a formal **access** type, whose `.all` / `new` / `null` are the interpreter's built-in access operations on the values (including **access-to-subprogram**: `Subprogram'Access` yields a first-class `Subprogram_Value` — a handle into the interpreter's subprogram-reference store — and a call through a name bound to one invokes the referenced subprogram); a formal **array** type, whose indexing / `'First` / `'Last` / `for...of` are the interpreter's built-in array operations; a formal **incomplete** type (`type T;`), which supports no operations, so the body only passes its values through; a formal **limited private** type (`type T is limited private;`), which (unlike plain private) has no `:=`/`=`, so the body uses only supplied formal subprograms; a formal **tagged private** type (`type T is tagged private;`), whose characteristic primitive operations are supplied as formal subprograms (different instances giving different implementations); a formal **abstract tagged private** type (`type T is abstract tagged private;`), whose abstract primitive is overridden concretely at instantiation; a formal **synchronized private** type (`type T is synchronized private;`), whose protected/task nature is unmodelled (erased); and a formal **abstract limited private** type (`type T is abstract limited private;`), combining the limited and abstract restrictions — so all the formal-type categories (private and its limited / tagged / abstract-tagged / synchronized / abstract-limited `Formal_Private_Type` flag variants, interface, derived, scalar, access, array, incomplete) run over the same erasure, their operations supplied as built-ins or as formal subprograms); a formal **package** binds (as a value) to the actual instance's `Package_Value`, so `P.Member` inside the template resolves through the actual instance's scope. Generic actuals are positional and **optional**: a formal beyond the supplied actuals falls back to its **default** (a formal object's `Default` expression, or a formal subprogram's `Default_Name`), raising only if a defaultless formal has no actual. An **`in out`** formal object (mode `Generic_In_Out`) aliases its actual variable, modelled per call like an `in out` subprogram parameter — copy-in at entry, copy-back to the actual on normal exit — so successive instance calls accumulate into the actual. A **subunit** body is modelled as completing an `is separate` stub: a `Subprogram_Name` whose `Completion` is a `Stub` raises a clear "is separate — its subunit is not compiled" error when invoked, and once the stub is completed in place (the `Completion` set to the subunit's `Block`, as `Diana.Library.Merge_Subunit` does on the library side) the call runs the subunit's body.

**Tasking and protected types** are executed in the sequential model their semantics reduce to single-threaded. `Elaborate` handles a `Protected_Body` by elaborating its operations (private state + protected subprograms) into a fresh scope and binding the object's name to a `Package_Value` — so a protected operation is a member call (`PO.Op`), a protected *procedure* mutates the shared state (`Assign` walks out to the protected scope), and a protected *function* reads it; mutual exclusion is trivially satisfied because the interpreter is sequential. A `Task_Body` is **activated** at the end of its enclosing declarative part (deferred during `Elaborate`, then run after all declarations): its body runs to completion in a fresh scope — a task's thread of control, modelled sequentially. (Member procedure calls — `PO.Op (...)` as a statement — are handled in `Execute` alongside the function-call member path.) A protected **entry** (an `Entry_Body`, registered by name during the protected object's elaboration) is called like a member procedure but first has its **barrier** evaluated in the protected object's scope: an open barrier runs the entry body, a closed one raises (a closed barrier would block forever in the sequential model, so it is an error rather than a wait). A task that offers **entries** keeps a persistent scope: at activation its local state is elaborated and its `accept` bodies are registered by entry name (`Env.Accepts`), and its name is bound to that scope; an `Entry_Call` (`Task.Entry (Actuals)`) is a **rendezvous** modelled synchronously — it runs the matching accept body with the accept's parameters bound to the call's actuals, in the task's scope. A **selective accept** (`select ... or ... end select`) registers each `Accept_Alternative`'s accept (and its `when` guard, in `Env.Accept_Guards`); an entry call to a guarded alternative checks the guard in the task's scope first, raising if it is closed (it would block).

A **generic package** is a `Generic_Name` whose `Specification` is a `Package_Specification` (rather than a `Generic_Subprogram_Header`). `Elaborate` instantiates one when it meets a `Package_Declaration` whose `Completion` is a `Unit_Instantiation`: `Instantiate_Package` creates a fresh **instance scope** (parented to the declaring scope, so a nested instance sees enclosing names), binds the generic formal objects to the actuals, elaborates the template's visible declarations into that scope, and binds the instance name to a `Package_Value` (a new `Static_Value` kind that is a runtime-only handle holding the instance scope's index). `Pkg.Member` resolves by looking the selector up in that scope; `Pkg.Sub (...)` calls the member subprogram with the instance scope as its static link (passed to `Invoke` as `Home`), so a member closes over the package's own generic formals and declarations.

A **child generic package** is instantiated against a parent *instance* — its `Generic_Instantiation.Generic_Unit` is a `Selected_Component` (`Parent_Instance.Child`) rather than a plain name. `Instantiate_Package` then nests the child's instance scope in the **parent instance's** scope (its `Home_Scope`) instead of the lexical declaring scope, so the child's declarations and member bodies see the parent's members by simple name (and the static-link walk from a child member reaches the parent's subprograms), exactly as a real child unit sees its parent's visible part.
